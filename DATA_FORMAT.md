# MCL Template Builder â€” Data Format Reference

This document explains the data model used by `mcltemplatebuilder.html` (a single-file
web app for laying out recurring MRI scanner schedules) so another Claude can read,
interpret, or generate its save files without opening the source.

`index.html` is the older sibling of this app; it uses a fixed 15-minute grid and the
storage key `mri-schedule-v1`. `mcltemplatebuilder.html` adds **per-scanner slot
durations** and uses storage key `mcl-template-builder-v1`. This doc describes the
newer file.

---

## Where the data lives

- **At runtime:** one global JS object named `data`, persisted to `localStorage` under
  the key `mcl-template-builder-v1`.
- **On export ("Save JSON"):** the entire `data` object, pretty-printed with
  `JSON.stringify(data, null, 2)`. The download is named `mri-schedule-<YYYY-MM-DD>.json`.
- **On import ("Load JSON"):** the file is parsed and passed through `migrate()`, which
  fills in any missing fields, then **replaces** all in-memory and localStorage state.

The JSON export is a **complete snapshot**, not a diff. There is no partial/merge format.

---

## Top-level shape

```json
{
  "settings":    { ... },
  "researchers": [ ... ],
  "schedules":   { ... }
}
```

### `settings`

```json
{
  "dayStartHour": 7,
  "dayEndHour": 21,
  "scannerNames": ["Prisma P1", "Prisma P2", "Varian 4T"],
  "scannerSlotMin": { "Prisma P1": 20, "Prisma P2": 20, "Varian 4T": 15 },
  "cycleStartDate": "2026-06-26",
  "sampleDataLoaded": true
}
```

| Field            | Meaning |
|------------------|---------|
| `dayStartHour`   | First hour shown on the grid (0â€“23). Applies to **all** scanners. |
| `dayEndHour`     | Last hour shown (exclusive end, 1â€“24). Must be `> dayStartHour`. |
| `scannerNames`   | Exactly 3 scanner names. These are the keys into `schedules` and `scannerSlotMin`. |
| `scannerSlotMin` | Per-scanner grid resolution in minutes (one of 10/15/20/30 in the UI). This is the single most important field for interpreting block times â€” see below. |
| `cycleStartDate` | The Monday that "Week 1" maps to. Used only by the ICS export to assign real calendar dates. `YYYY-MM-DD`. |
| `sampleDataLoaded` | Internal flag. When false/absent, `migrate()` injects 20 demo studies. Leave `true` to avoid re-seeding. |

### `researchers`

A flat list of **studies** (the term "researcher" is legacy; each entry is really a study
owned by a PI). These are **global** â€” shared across all scanners and weeks. The left-hand
palette renders this list.

```json
{
  "id": "ab12cd34",
  "name": "Aging & Memory",
  "short": "Aging",
  "pi": "Anderson",
  "duration": 60,
  "color": "#dc2626",
  "notes": ""
}
```

| Field      | Meaning |
|------------|---------|
| `id`       | Unique string, generated by `uid()` (random base-36 + timestamp suffix). Referenced by blocks via `researcherId`. |
| `name`     | Full study name. |
| `short`    | Abbreviated label shown on calendar blocks. Falls back to `name` if empty. |
| `pi`       | Principal investigator. Used for grouping/filtering in the block-edit modal. |
| `duration` | **Default** block length in minutes when dragged onto the calendar. Stored in real minutes (not slots); rounded to the target scanner's grid on drop. UI allows 5-min steps. |
| `color`    | Hex color for the block and the palette card's left border. |
| `notes`    | Free text. |

### `schedules`

A nested map: **scanner name â†’ week number (1â€“4) â†’ array of blocks**.

```json
{
  "Prisma P1": {
    "1": [ <block>, <block>, ... ],
    "2": [ ... ],
    "3": [ ... ],
    "4": [ ... ]
  },
  "Prisma P2": { "1": [...], "2": [...], "3": [...], "4": [...] },
  "Varian 4T": { "1": [...], "2": [...], "3": [...], "4": [...] }
}
```

- The three top-level keys **must** match `settings.scannerNames`.
- Week keys are the strings `"1"`â€“`"4"` (a 4-week repeating cycle).
- Each week is an unordered array of block objects.

A **block** = one scheduled scan slot:

```json
{
  "id": "ef56gh78",
  "researcherId": "ab12cd34",
  "day": 0,
  "startSlot": 6,
  "durationSlots": 3,
  "label": "",
  "notes": ""
}
```

| Field           | Meaning |
|-----------------|---------|
| `id`            | Unique block id (`uid()`). |
| `researcherId`  | Foreign key into `researchers[].id`. May be `null` for a blank/manually-created block. If it points to a deleted study, the block renders as "(untitled)". |
| `day`           | Day of week as an integer **0â€“6 = Mondayâ€“Sunday** (note: Monday is 0, not Sunday). |
| `startSlot`     | Start position as a **count of grid rows** from `dayStartHour`, **not** a clock time. |
| `durationSlots` | Length as a **count of grid rows**, **not** minutes. |
| `label`         | Optional per-block label override. If empty, the displayed name falls back to the study's `short`, then `name`. |
| `notes`         | Optional per-block notes (separate from the study's notes). |

## How to convert a block to a real time  âš ď¸Ź critical

Block positions are stored in **grid rows**, and **each scanner has its own row size**
(`scannerSlotMin`). The same `startSlot`/`durationSlots` therefore mean *different clock
times on different scanners*. To resolve a block, you must know which scanner's array it
came from.

```
slotMin   = settings.scannerSlotMin[scannerName]          // e.g. 20 for Prisma P1, 15 for Varian 4T
startMin  = settings.dayStartHour * 60 + startSlot * slotMin
endMin    = startMin + durationSlots * slotMin
durationMinutes = durationSlots * slotMin
```

Convert `startMin` / `endMin` (minutes past midnight) to a clock time as usual.

**Worked example** â€” the block above (`startSlot: 6`, `durationSlots: 3`):

- On **Prisma P1** (`slotMin = 20`): start = 7:00 + 6Ă—20 = **9:00 am**, duration = 3Ă—20 = **60 min**, end = **10:00 am**.
- On **Varian 4T** (`slotMin = 15`): start = 7:00 + 6Ă—15 = **8:30 am**, duration = 3Ă—15 = **45 min**, end = **9:15 am**.

So **never interpret a block without first looking up its scanner's `scannerSlotMin`.**

## The 4-week cycle and the ICS export

The schedule is a **repeating 4-week template**, not dated events. Weeks 1â€“4 cycle
indefinitely. The "Export ICS" feature is what materializes real calendar dates:

- It asks how many weeks to generate, then walks `w = 0, 1, 2, â€¦`.
- For each generated week it picks template week `(w % 4) + 1`.
- The calendar date = `cycleStartDate + w*7 + day` days.
- Times come from the per-scanner `scannerSlotMin` math above.

The JSON itself contains **no absolute dates or clock times** â€” only the cycle template
plus `cycleStartDate`. If you need a human-readable calendar, generate it from the ICS
logic, not from the raw JSON.

## Invariants / gotchas for anyone editing the JSON

- `schedules` keys, `scannerSlotMin` keys, and `scannerNames` must all be the **same 3
  scanner names**. Renaming a scanner means renaming it in all three places (the Settings
  UI does this and preserves the schedule data by index).
- Every scanner must have all four week keys `"1"`â€“`"4"` present (may be empty arrays).
- `day` is **Monday-indexed** (0 = Monday).
- `startSlot` + `durationSlots` should fit within the grid:
  `startSlot + durationSlots â‰¤ (dayEndHour - dayStartHour) * 60 / slotMin`. The app clamps
  this on edit but won't repair an imported file.
- `researcherId` should reference an existing `researchers[].id`, or be `null`.
- `duration` on a study is in **minutes**; `durationSlots` on a block is in **rows**. Don't
  confuse them.
- Changing a scanner's `scannerSlotMin` in the UI re-quantizes its existing blocks to keep
  clock times stable (`convertBlockSlotMin`). If you change `scannerSlotMin` by hand in the
  JSON **without** re-scaling `startSlot`/`durationSlots`, every block on that scanner will
  shift in time.
